# AngelCRM 缺口与反人类设计审计

更新时间：2026-07-07

## 1. 当前最缺的能力

### 1.1 真实用户与账号体系

当前所有用户共用 `demo2026` 登录，用户没有独立密码、密码重置、首次登录改密、停用后 token 失效等机制。对于 CRM 来说，这是生产化第一短板。

建议：

- 用户表增加 `passwordHash`。
- 超管创建用户时生成临时密码或邀请链接。
- 首次登录强制改密。
- 停用用户后立即拒绝旧 token。
- 登录失败限流。

### 1.2 真实系统管理后台

设置页现在更像“静态展板”，角色、部门、市场、通知、模板、账号资料基本还没有完整保存逻辑。

建议拆成：

- 用户管理
- 角色权限
- 市场与部门
- 年度目标
- 文档模板
- 通知策略
- 审计日志
- 数据迁移
- 个人账号设置

### 1.3 业务领域 API

通用 CRUD 已经加了权限，但 CRM 不能长期停留在“表格增删改查”。业务动作需要领域 API：

- 线索转化：创建客户、联系人、商机，不只是把 lead 标记 converted。
- 商机阶段推进：需要状态机和输赢单原因。
- 合同审批：法务流转、到期提醒。
- 订单履约：PI、PO、发货、报关、完成。
- 收款确认：Finance 专属确认、凭证、逾期。

### 1.4 数据库与数据一致性

JSON 文件适合原型，不适合多人 CRM。现在没有外键、唯一约束、事务、索引、分页查询。

建议尽快落 PostgreSQL，并复用 `backend/docs/schema.sql` 的方向。

## 2. 明显反人类设计

### 2.1 Settings 页面塞太多高危功能

品牌规范、角色、审计、账号设置、数据迁移全塞在一个 Tabs 里。普通用户很难理解哪个是“个人设置”，哪个是“系统级危险操作”。

改法：

- 个人账号设置独立出来。
- 系统设置只给超管。
- 数据迁移放独立页面，使用强确认和预检。

### 2.2 迁移包导入过于直接

当前流程是选文件后确认覆盖，缺少“预检报告”。用户不知道会覆盖哪些表、多少条、版本是否兼容。

改法：

- 上传后先 validate。
- 显示实体清单、记录数、导出人、导出时间。
- 要求输入确认短语。
- 自动备份当前数据。
- 导入后显示报告。

### 2.3 顶部“新建”入口和页面能力曾经不一致

顶部新建跳到页面 `?create=1`，但如果角色无权限，用户会点击后才失败。现在已开始按权限隐藏，但各页面内部按钮还需要继续做按钮级权限。

改法：

- 所有页面的创建、编辑、删除、转换、确认按钮都接 `canCreate/canUpdate/canDelete`。
- 后端仍作为最终防线。

### 2.4 删除入口太轻

客户卡片里删除藏在三点菜单，没有二次确认，也没有软删除。CRM 里客户、合同、订单都不应该随手硬删。

改法：

- 删除改为停用 / 归档 / 作废。
- 高危删除二次确认。
- 审计记录删除原因。
- 超管才可物理删除。

### 2.5 报表和列表混用 mock 与真实数据

同一个系统里有些页面读后端，有些页面读 `crmData.ts`，用户会看到“明明刚新增了，报表不变化”的割裂体验。

改法：

- 报表统一后端聚合 API。
- `crmData.ts` 只保留类型和 seed 参考，页面不直接读 mock 数据。

### 2.6 表单缺少业务默认与上下文

例如创建客户 owner 下拉写死 `u2/u3/u4/u5`；线索转化并没有真正创建 account/contact/opportunity；创建订单的客户和产品依赖全量数据。

改法：

- owner 从后端用户 API 按角色过滤。
- 线索转化做完整事务。
- 表单默认值来自当前用户、市场、所处页面上下文。

### 2.7 权限失败反馈不足

以前无权限直接跳首页或请求失败，用户不知道发生了什么。现在已补基础 403 页面，但接口错误还需要统一中文业务提示。

改法：

- axios 拦截 403 显示“无权限操作”。
- 表单写失败保留用户输入，不要直接关闭弹窗。
- 高危失败给出原因和联系管理员入口。

## 3. 下一阶段优先级

### P0 已开始处理

- 迁移包鉴权。
- 迁移包审计。
- 系统管理入口超管可见。
- 更新脚本不依赖裸 API。

### P1 继续推进

- 用户独立密码和停用校验。
- 按钮级权限。
- 读数据范围继续细化。
- 审计日志查询真实化。
- 删除改软删除。

### P2 再推进

- 系统设置拆页面。
- 领域 API 替代通用 CRUD。
- PostgreSQL 数据层。
- E2E 测试和后端 API 测试。

## 4. 当前工程判断

这个项目已经不是“页面没做完”的问题，而是需要从 demo 系统转成运营系统。

最该避免的方向：

- 继续堆页面。
- 继续在前端 mock 上做报表。
- 继续把系统管理塞进 Settings。
- 继续用通用 CRUD 冒充业务流程。

最该优先做的方向：

- 权限。
- 审计。
- 用户账号。
- 数据一致性。
- 领域流程。
- 真实报表。
